### 2.1 Injury Detection and Prediction

#### What Data Is Needed

| Data Category       | Specific Metrics                                                                                  | Collection Method                                              |
| ------------------- | ------------------------------------------------------------------------------------------------- | -------------------------------------------------------------- |
| Training Load       | Session RPE (rate of perceived exertion), duration, intensity zones, total distance, sprint count | Manual input via app (RPE), wearable integration (distance/HR) |
| Sleep & Recovery    | Sleep duration, sleep quality, HRV (heart rate variability), resting heart rate                   | Apple Health / Google Fit / Oura / Whoop API                   |
| Biomechanical       | Asymmetry scores, ground contact time, stride length                                              | Wearable IMU sensors (future integration)                      |
| Historical Injury   | Previous injuries, type, severity, time to return                                                 | Self-reported in profile + medical forms                       |
| Workload Ratios     | Acute:Chronic Workload Ratio (ACWR), training monotony, training strain                           | Calculated from training load time-series                      |
| Subjective Wellness | Muscle soreness, fatigue, mood, stress, hydration                                                 | Daily wellness questionnaire in-app                            |
| Match Load          | Minutes played, position-specific demands, travel schedule                                        | Manual input or integration with league APIs                   |

#### Models That Work

1. **Acute:Chronic Workload Ratio (ACWR) Model**
   - Not ML per se, but the foundational metric. ACWR = (acute load / chronic load), where acute = 7-day rolling average, chronic = 28-day rolling average.
   - "Sweet spot" for injury risk: ACWR between 0.8-1.3. Above 1.5 = high risk.
   - Simple to implement as a first pass.

2. **Random Forest / Gradient Boosted Trees (XGBoost, LightGBM)**
   - Best performing in most sports injury prediction literature.
   - Features: ACWR, cumulative load, sleep hours, HRV, previous injury history, age, position.
   - Handles mixed feature types well, interpretable with SHAP values.
   - Typical AUC: 0.65-0.78 in research studies (injury prediction is inherently noisy).

3. **Time-Series Models (LSTM, Temporal Convolutional Networks)**
   - Capture patterns over time (e.g., gradual overload trends).
   - Input: Sliding window of 14-28 days of daily metrics.
   - Better for detecting "creeping" overtraining patterns.
   - More complex to train, need more data per athlete.

4. **Bayesian Networks**
   - Good for small sample sizes (common in sports).
   - Can incorporate prior knowledge (e.g., "hamstring injuries are more likely after 3+ days of high sprint load").
   - Interpretable outputs (probability chains).

| Platform                          | What It Offers                                                                  | Integration Complexity                                    |
| --------------------------------- | ------------------------------------------------------------------------------- | --------------------------------------------------------- |
| **Kitman Labs**                   | Injury risk analytics, performance optimization. Enterprise SaaS for pro teams. | High (enterprise contracts, not individual-facing)        |
| **Strive (now part of Catapult)** | Wearable sensors + injury risk dashboards                                       | Hardware-dependent; API not publicly available            |

#### What's Feasible for OnlySports

**MVP approach**: Don't build full CV pipeline from scratch. Instead:

1. **Ball tracking + player tracking** using pre-trained YOLOv8 models fine-tuned on sports footage
2. **Pose estimation** using MediaPipe BlazePose (runs on-device) for form analysis
3. **Basic stat extraction**: shots on goal (soccer), shooting attempts (basketball), serve speed approximation (tennis) using trajectory analysis
4. **Integration with manual stat input** enhanced by AI suggestions

**Cloud processing** is required for video analysis at any meaningful quality. On-device is only feasible for real-time pose estimation on single-person frames.

### 2.3 Player Performance Scoring

#### Composite Metric Design

**Universal Athlete Score (UAS)**: A 0-100 composite score that accounts for:

1. **Physical Metrics** (30% weight)
   - Sport-specific physical benchmarks (speed, endurance, strength)
   - Trend direction (improving/declining over time)
   - Percentile ranking within age group and sport

2. **Technical/Skill Metrics** (35% weight)
   - Position-specific stats (goals, assists, tackles, saves, etc.)
   - Efficiency metrics (completion %, accuracy %, success rate)
   - Consistency score (standard deviation of performance)

3. **Activity & Engagement** (15% weight)
   - Training frequency and load
   - Competition participation rate
   - Platform engagement (posting highlights, updating stats)

4. **Growth Trajectory** (20% weight)
   - Rate of improvement over 30/60/90 days
   - Predicted performance curve (regression-based)
   - Milestone achievements

#### Position-Specific Ratings

Each sport/position gets a tailored stat weighting. Examples:

**Soccer - Striker**: Goals (25%), xG performance (20%), shot accuracy (15%), key passes (10%), aerial duels won (10%), pressing actions (10%), minutes played (10%)

**Basketball - Point Guard**: Assists (20%), assist-to-turnover ratio (15%), PER (15%), 3PT% (10%), steals (10%), plus-minus (15%), usage rate (15%)

**Tennis**: Win rate (25%), serve % (15%), break point conversion (15%), ranking trajectory (20%), tournament level (15%), head-to-head record (10%)

#### Model Approach

- **Weighted composite scoring** for the base metric (interpretable, adjustable)
- **Bayesian updating** for incorporating new data points without full recalculation
- **Percentile normalization** within cohorts (age, sport, level) to ensure fairness
- **Time-decay weighting**: Recent performances weighted more heavily (exponential decay, half-life = 60 days)

### 2.4 AI Scouting Recommendations

#### Player-Team Matching Algorithm

**Approach**: Multi-factor recommendation system combining:

1. **Content-Based Filtering**
   - Match player attribute vectors to team need vectors
   - Features: position, physical metrics, performance stats, age, location, competition level
   - Cosine similarity between player profile embedding and team requirement embedding

2. **Collaborative Filtering**
   - "Scouts who viewed Player A also viewed Player B"
   - "Teams that recruited Player X from this profile type also recruited Player Y"
   - Matrix factorization (ALS or SVD) on scout-player interaction matrix

3. **Hybrid Model**
   - Combine content-based and collaborative signals
   - Add contextual features: timing (transfer windows), geographic preferences, budget tier
   - LightFM library is specifically designed for this hybrid approach

#### Data Signals for Scouting

| Signal                             | Source                                         | Weight |
| ---------------------------------- | ---------------------------------------------- | ------ |
| Performance stats match team needs | Player profile + team requirements             | High   |
| Profile view patterns              | App analytics (who views whom)                 | Medium |
| Communication patterns             | Message initiation between scouts and athletes | High   |
| Geographic proximity               | Location data                                  | Medium |
| Competition level compatibility    | League/level metadata                          | High   |
| Age and development stage          | Profile data                                   | Medium |
| Social proof                       | Endorsements, follower count among scouts      | Low    |

#### Implementation

- **Embedding model**: Train a simple neural network to create 64-dimensional embeddings for both players and teams/scouts
- **Similarity search**: Use pgvector (Postgres extension, available on Supabase) for efficient nearest-neighbor search
- **Re-ranking**: Apply business rules (geographic limits, sport match, etc.) after initial retrieval

### 2.5 Content Recommendation Engine

#### Feed Personalization Design

**Multi-signal ranking model** for the content feed:

1. **Engagement Prediction Score** (primary signal)
   - Predict P(like), P(comment), P(share), P(watch_complete) for each post-user pair
   - Features: user sport interest, user role, post type, author relationship, post age, post engagement rate
   - Model: Gradient boosted trees (LightGBM) or logistic regression for MVP

2. **Interest Matching Score**
   - User interest vector (derived from: stated sport preferences, engagement history, followed users' sports)
   - Post topic vector (sport tags, post type, content analysis)
   - Dot product similarity

3. **Social Graph Score**
   - Posts from followed users weighted higher
   - Posts from users in same sport/position weighted higher
   - "Friend of friend" posts given moderate boost

4. **Freshness Decay**
   - Exponential time decay (half-life = 6 hours for social content, 48 hours for opportunities)

5. **Diversity Injection**
   - Ensure feed mixes content types (highlights, training, opportunities, achievements)
   - Exploration vs exploitation: 80% personalized, 20% discovery/trending

#### Cold Start Problem Solutions

- New users: Use role + sport selection from onboarding to bootstrap recommendations
- New posts: Initially distribute to followers + same-sport users, then let engagement signals take over
- Popularity-based fallback for very new users (trending content in their sport)

### 2.6 Injury Risk Assessment from Training Patterns

#### Workload Monitoring System

**Core Metrics to Track**:

1. **sRPE (Session Rating of Perceived Exertion)**
   - Athlete rates session intensity 1-10, multiplied by duration in minutes
   - Example: 7 intensity x 90 min = 630 AU (arbitrary units)
   - Simplest and most validated training load metric in sports science

2. **ACWR (Acute:Chronic Workload Ratio)**
   - Acute = 7-day exponentially weighted moving average of sRPE
   - Chronic = 28-day exponentially weighted moving average of sRPE
   - EWMA preferred over rolling averages (accounts for decaying fitness/fatigue)
   - Risk zones: <0.8 (undertrained), 0.8-1.3 (optimal), 1.3-1.5 (caution), >1.5 (danger)

3. **Training Monotony**
   - Mean daily load / standard deviation of daily load (over 7 days)
   - High monotony (>2.0) indicates repetitive training that increases injury risk

4. **Training Strain**
   - Weekly load x monotony
   - Values above 5,000-6,000 AU associated with increased illness risk

5. **Week-to-Week Load Change**
   - > 10% increase week-over-week is a risk factor
   - "10% rule" widely used in endurance sports

#### Alert System Design

| Alert Level | Trigger                                                                     | User-Facing Message                                                                    |
| ----------- | --------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| Green       | ACWR 0.8-1.3, monotony <1.5                                                 | "Training load is optimal. Keep it up!"                                                |
| Yellow      | ACWR 1.3-1.5 OR monotony 1.5-2.0 OR >10% weekly increase                    | "Your training load has increased significantly. Consider a lighter session tomorrow." |
| Orange      | ACWR >1.5 OR monotony >2.0 OR insufficient sleep (<6h avg)                  | "High injury risk detected. We recommend a recovery day."                              |
| Red         | ACWR >1.5 AND (poor sleep OR high monotony OR previous injury in same area) | "Warning: Multiple risk factors elevated. Strongly recommend rest and recovery."       |

### 3.1 Architecture Overview: React Native + Supabase ML Stack

```
React Native App (Expo)
    |
    |-- Apple HealthKit / Google Fit (on-device data)
    |-- MediaPipe (on-device pose estimation)
    |-- TensorFlow Lite (on-device inference for simple models)
    |
    v
Supabase
    |-- PostgreSQL (data storage, pgvector for embeddings)
    |-- Edge Functions (Deno/TypeScript - lightweight ML logic, API orchestration)
    |-- Realtime (push notifications for alerts)
    |-- Storage (video/image assets)
    |
    v
External ML Services (cloud)
    |-- Google Cloud Vision / Vertex AI (video analysis)
    |-- OpenAI API (content understanding, text embeddings)
    |-- Custom ML API (Python FastAPI on Cloud Run / Railway)
```

**Key Architectural Decision**: Supabase Edge Functions are limited to Deno/TypeScript runtime. For anything requiring Python ML libraries (scikit-learn, TensorFlow, PyTorch), you need a separate Python microservice. Edge Functions are ideal for orchestration, simple calculations, and API proxying.

### 3.2 Feature: Injury Risk Assessment from Training Patterns

**Priority**: HIGH (unique differentiator, achievable with minimal ML)

#### MVP Approach (Weeks 1-4)

**Data Sources**:

- Manual daily wellness check-in (RPE, soreness, sleep hours, mood) -- collected in-app
- Training session logs (type, duration, intensity) -- manual input
- Apple HealthKit / Google Fit data (sleep, resting HR, steps) -- automated

**Implementation**:

- All calculations in **Supabase Edge Functions** (no external ML needed for MVP)
- ACWR, monotony, strain are simple rolling-window calculations
- Store daily load entries in a `training_logs` table
- Edge Function triggered on new log entry; computes ACWR and triggers alerts
- Risk level stored in `athlete_risk_status` table
- Push notification via Supabase Realtime when risk level changes

**Schema additions**:

```sql
CREATE TABLE training_logs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id),
  date DATE NOT NULL,
  session_type TEXT, -- 'training', 'match', 'gym', 'recovery'
  duration_minutes INT,
  rpe INT CHECK (rpe BETWEEN 1 AND 10),
  srpe INT GENERATED ALWAYS AS (duration_minutes * rpe) STORED,
  notes TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE wellness_logs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id),
  date DATE NOT NULL,
  sleep_hours DECIMAL(3,1),
  sleep_quality INT CHECK (sleep_quality BETWEEN 1 AND 5),
  muscle_soreness INT CHECK (muscle_soreness BETWEEN 1 AND 5),
  fatigue INT CHECK (fatigue BETWEEN 1 AND 5),
  mood INT CHECK (mood BETWEEN 1 AND 5),
  stress INT CHECK (stress BETWEEN 1 AND 5),
  hydration INT CHECK (hydration BETWEEN 1 AND 5),
  created_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE injury_risk_alerts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id),
  date DATE NOT NULL,
  acwr DECIMAL(4,2),
  monotony DECIMAL(4,2),
  strain DECIMAL(8,1),
  risk_level TEXT CHECK (risk_level IN ('green', 'yellow', 'orange', 'red')),
  message TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);
```

**React Native Integration**:

- `react-native-health` (iOS HealthKit) / `react-native-google-fit` (Android)
- Daily check-in modal (takes 30 seconds; 5 sliders + duration/RPE)
- Dashboard widget showing ACWR gauge, risk level, trend chart
- Push notifications for risk level changes

**Runs on**: Supabase Edge Functions (Deno). No external ML service needed.

**Timeline**: 4 weeks for full MVP

#### Full Implementation (Months 2-3)

- Add wearable integrations (Whoop API, Oura API, Garmin Health API)
- Train Random Forest injury prediction model on aggregated (anonymized) user data
- Deploy Python model on Cloud Run; Edge Function proxies requests
- Personalized risk thresholds based on individual athlete history
- Integrate with calendar for rest day recommendations

**Priority**: HIGH (core value prop for athlete profiles)

**Data Sources**:

- Self-reported stats (manual entry per game/season)
- Training log data (frequency, consistency)

**Implementation**:

- **Supabase Edge Functions** for score calculation
- Weighted composite formula (no ML needed initially)

- Score recalculated on new stat entry (triggered by database webhook)

**Schema additions**:

```sql
CREATE TABLE player_stats (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

CREATE TABLE performance_scores (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES profiles(id),
  sport TEXT NOT NULL,
  position TEXT,
  overall_score DECIMAL(5,2),
  physical_score DECIMAL(5,2),
  technical_score DECIMAL(5,2),
  consistency_score DECIMAL(5,2),
  growth_score DECIMAL(5,2),
  percentile_rank DECIMAL(5,2),
  calculated_at TIMESTAMPTZ DEFAULT now()
);
```

**Runs on**: Supabase Edge Functions. Percentile calculations via SQL window functions.

**Timeline**: 4-5 weeks

#### Full Implementation (Months 3-5)

- Bayesian updating for scores as new data arrives
- ML-based growth trajectory prediction (linear regression, then gradient boosted)
- Position-specific detailed breakdowns
- Peer comparison visualizations
- Verified stats through coach/team confirmation workflow

### 3.4 Feature: Content Recommendation Engine

**Priority**: MEDIUM-HIGH (improves engagement and retention)

#### MVP Approach (Weeks 3-6)

**Data Sources**:

- User profile (sport, role, position)
- Social graph (follows, blocks)
- Engagement history (likes, comments, time spent)
- Post metadata (sport tags, post type, author)

**Implementation**:

- **Phase 1 (Supabase-only)**: Rule-based ranking in a Supabase Edge Function
  - Score = (is_followed _ 5) + (same_sport _ 3) + (same_role _ 1) + (engagement_rate _ 2) - (age_hours \* 0.1)
  - Query with ORDER BY calculated score
  - Cache ranked feed in a `feed_cache` table, refresh every 15 minutes

- **Phase 2**: Simple embedding-based approach
  - Use OpenAI `text-embedding-3-small` API to embed post captions and user bios
  - Store embeddings in pgvector column
  - Nearest-neighbor retrieval for "posts similar to what you've liked"
  - Supabase has native pgvector support

**Runs on**: Supabase Edge Functions + pgvector. OpenAI API for embeddings (optional Phase 2).

**Timeline**: 3 weeks for Phase 1, +3 weeks for Phase 2

#### Full Implementation (Months 3-6)

- Train a LightGBM engagement prediction model on user-post interaction data
- Deploy on Cloud Run Python service
- A/B testing framework for ranking algorithms
- Real-time re-ranking based on session behavior
- Multi-armed bandit for exploration vs exploitation

### 3.5 Feature: AI Scouting Recommendations

**Priority**: MEDIUM (high value but needs critical mass of users)

#### MVP Approach (Weeks 6-10)

**Data Sources**:

- Player profiles (sport, position, stats, location, competition level)
- Scout/team profiles (sport, positions needed, location)
- Opportunity listings (requirements, sport, level)
- Interaction data (profile views, messages sent, opportunity applications)

**Implementation**:

- **Content-based matching** in Supabase Edge Functions
  - Create feature vectors for players and opportunities
  - Cosine similarity matching using pgvector
  - "Recommended Athletes" for scouts based on their search history and team needs
  - "Recommended Opportunities" for athletes based on their profile match

**Schema additions**:

### 3.6 Feature: Match Stats Tracking (Computer Vision)

**Priority**: LOW-MEDIUM (high complexity, impressive but hard to execute well)

#### MVP Approach (Weeks 8-14)

**Data Sources**:

- User-uploaded game footage (via Supabase Storage)
- Manual stat input (primary method, enhanced by AI)

**Implementation**:

- **Phase 1: Enhanced manual stat tracking** (no CV)
  - In-app game stat tracker (tap-based: "shot", "goal", "assist", "foul")
  - Real-time stat entry during games
  - Post-game stat summaries and visualizations
  - This alone is valuable and builds the data foundation

- **Phase 2: Basic CV features** (on-device)
  - **MediaPipe Pose Estimation** in React Native via `react-native-mediapipe` or `expo-camera` + TensorFlow.js
  - Use cases: form analysis for individual training videos (shooting form, serve motion, running gait)
  - On-device processing; no cloud costs
  - NOT full game analysis -- just individual technique clips

- **Phase 3: Cloud video analysis** (future)
  - Upload game footage to cloud processing pipeline
  - Google Video Intelligence API for scene/shot detection
  - Custom YOLOv8 model for ball/player detection (trained on sport-specific data)
  - Extract basic stats: possession time, shot count, player positions
  - Significant cost per video (~$0.50-$2.00 per minute of footage)

**React Native Libraries**:

- `@mediapipe/tasks-vision` (web/JS port)
- `react-native-vision-camera` + frame processor plugins
- `@tensorflow/tfjs-react-native` for on-device inference

**Runs on**: Phase 1-2 on-device + Supabase. Phase 3 requires Google Cloud / custom Python service.

**Timeline**: Phase 1: 3 weeks. Phase 2: 4 weeks. Phase 3: 8-12 weeks (significant R&D).

### 3.7 Feature: Injury Detection from Biomechanics (Advanced)

**Priority**: LOW (requires specialized hardware data; long-term differentiator)

#### MVP Approach

- Leverage pose estimation (MediaPipe) to detect asymmetry in movement
- Compare left vs right side joint angles during exercises
- Flag significant asymmetry (>15% difference) as a potential injury risk factor
- Requires the athlete to record themselves performing specific movements (squat, lunge, single-leg hop)

#### Full Implementation

- Integration with IMU wearables (accelerometers, gyroscopes)
- Biomechanical model using LSTM on time-series joint angle data
- Gait analysis from video (requires calibrated camera setup)
- Partnership with sports science institutions for validated models

**Timeline**: MVP 6-8 weeks (after pose estimation is working). Full: 6-12 months.

### 3.11 Key Risks and Mitigations

| Risk                                     | Mitigation                                                                                                                     |
| ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| Injury prediction liability (legal)      | Frame as "wellness insights", not medical advice. Disclaimer required. Never say "you will get injured."                       |
| Cold start for recommendations           | Aggressive onboarding (sport, position, interests). Rule-based fallback. Trending content for new users.                       |
| Low data quality (self-reported stats)   | Coach/team verification workflows. Gamify consistent logging. Cross-validate with wearable data.                               |
| Video CV accuracy too low for production | Start with manual stat tracking. CV is enhancement, not replacement. Phase in gradually.                                       |
| pgvector performance at scale            | 100K profiles with 64-dim vectors is trivial for pgvector. Only becomes a concern at 1M+. Use IVFFlat index.                   |
| User privacy concerns with health data   | Clear consent flows. Data stored encrypted. User can delete anytime. GDPR/CCPA compliance. Never share individual health data. |
| Cost of cloud video processing           | Gate video analysis behind premium tier. Limit to X minutes/month per user.                                                    |
